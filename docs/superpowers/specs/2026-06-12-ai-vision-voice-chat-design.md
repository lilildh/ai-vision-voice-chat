# AI 视觉语音对话应用设计文档

## 1. 项目定位

本项目计划实现一款 Web 浏览器应用。用户打开摄像头和麦克风后，可以通过语音向 AI 提问，AI 结合用户说的话和摄像头画面中的关键帧内容进行理解，并以文本和语音给出回应。

首版不追求全实时视频通话式 AI，而采用“语音轮次 + 视频关键帧理解”的混合模式：用户说完一轮问题后，前端截取当前视频关键帧，连同语音转写文本发送给后端；后端调用真实云端多模态模型；前端展示 AI 文本回复，并用浏览器 TTS 播报。

应用主场景聚焦“桌面/室内场景问答与即时指导”，例如识别桌面物体、解释摄像头里看到的内容、根据画面给出下一步建议、回答关于当前画面的追问。

## 2. 已确认的设计决策

- 首版形态：Web 浏览器应用。
- 交互模式：语音轮次 + 视频关键帧理解。
- 场景范围：桌面/室内场景问答与即时指导。
- 模型策略：真实接入云端 AI，通过后端 provider 适配层隔离供应商。
- ASR 策略：优先使用浏览器 Web Speech API，云端 ASR 作为预留接口。
- TTS 策略：文本即时显示，浏览器 `speechSynthesis` 播报，云端 TTS 作为预留接口。
- 视觉采样：每轮默认上传 1 张关键帧，必要时最多 3 张。
- 主动观察：首版不做 AI 主动观察或打断。
- 记忆策略：保留短期会话记忆，不做跨会话长期记忆。
- 隐私策略：默认不持久化原始音频、视频或图片。
- 账号计费：首版不做登录、账号和计费系统，只做演示级限流与统计。
- UI 布局：左右分栏工作台。
- 技术栈：Vite + React + TypeScript 前端，Node/Express 后端。
- 模型适配：OpenAI-compatible 多模态 provider，使用环境变量配置服务地址、模型名和 API Key。
- 成本控制：请求级估算 + 会话级预算 + 服务端硬限制。

## 3. 目标用户与核心场景

### 3.1 目标用户

首版面向需要快速体验多模态交互能力的普通用户、产品评审者和开发演示场景。用户不需要理解模型接口，只需要允许浏览器访问摄像头和麦克风，就可以进行“看图说话式”的 AI 对话。

### 3.2 典型场景

- 用户把桌面、纸张、屏幕或手持物体对准摄像头，询问“你看到了什么”。
- 用户继续追问“左边那个是什么颜色”“这个界面下一步该点哪里”。
- 用户展示一张纸或一个物体，请 AI 给出概括性说明。
- 用户希望 AI 在看不清时明确说明原因，并提示如何调整摄像头或光线。

### 3.3 非目标场景

首版不做以下高风险或高复杂度能力：

- 医疗诊断、安防监控、驾驶辅助、身份识别、情绪判断。
- 连续视频流实时理解。
- AI 主动观察、主动打断或后台持续监控。
- 长期用户画像、跨会话记忆或个性化学习。
- 专业级 OCR、专业图像检测或复杂工业视觉判断。
- 登录、套餐、余额、商业账单系统。

## 4. 用户故事

### 4.1 计划实现的用户故事

1. 作为用户，我可以在浏览器中启动摄像头和麦克风，让应用看到当前画面并听到我的提问。
2. 作为用户，我可以用自然语言询问摄像头里的内容，例如“你看到桌上有什么”。
3. 作为用户，我可以在同一会话中追问，例如“左边那个是什么颜色”，AI 能结合短期上下文和当前画面回答。
4. 作为用户，我可以看到语音转写文本、AI 文本回复和当前系统状态。
5. 作为用户，我可以听到 AI 用语音播报回复，也可以中止播报。
6. 作为用户，当浏览器权限、ASR、模型调用或限流失败时，我能看到明确错误提示。
7. 作为用户，我可以看到本轮请求使用了几张关键帧、耗时多久、估算成本是多少。
8. 作为开发者，我可以在后端替换不同云端多模态模型，而不改动前端主流程。
9. 作为开发者，我可以通过环境变量配置模型服务地址、模型名和 API Key，避免在浏览器暴露密钥。
10. 作为产品评审者，我可以通过一组桌面/室内测试任务判断视觉理解是否可靠、语音交互是否流畅、成本控制是否生效。

### 4.2 首版承诺实现的用户故事

首版必须实现以下用户故事：

- 打开摄像头并显示实时预览。
- 打开麦克风并通过浏览器 Web Speech API 生成语音转写。
- 用户说完一轮问题后，截取 1 张默认关键帧；当画面变化明显或用户要求看清楚时，最多补充到 3 张。
- 将语音文本和关键帧发送到后端。
- 后端通过真实云端多模态 provider 获取 AI 文本回复。
- 前端展示用户文本、AI 文本、系统状态、错误提示。
- 前端用浏览器 TTS 播报 AI 回复，并支持停止播报。
- 维护当前页面内的短期会话上下文。
- 展示请求次数、关键帧数量、延迟和估算成本。
- 服务端执行每轮关键帧上限、请求频率上限、会话轮次上限和图片压缩策略。

### 4.3 暂不实现的用户故事

以下用户故事属于后续版本：

- 用户登录、账户、套餐、计费和多用户隔离。
- 云端 ASR 和云端流式 TTS 默认启用。
- 连续视频流上传和全实时视频通话式 AI。
- AI 主动观察画面并主动提醒用户。
- 保存原始截图、录音、录像或跨会话历史。
- 专业 OCR、专业图像检测和高风险领域判断。

## 5. 产品交互设计

### 5.1 页面布局

首版采用左右分栏工作台布局。

左侧为摄像头区域：

- 显示实时摄像头预览。
- 显示摄像头权限状态。
- 显示当前采样状态，例如“待机”“准备取帧”“已取 1 张关键帧”。
- 显示画面质量提示，例如过暗、模糊、摄像头不可用。

右侧为对话区域：

- 显示用户语音转写文本。
- 显示 AI 文本回复。
- 显示 TTS 播报状态。
- 支持停止当前播报。
- 显示错误和恢复建议。

顶部或底部为控制与统计区域：

- 开始/停止会话。
- 麦克风状态。
- 摄像头状态。
- 当前模型状态。
- 本轮延迟。
- 请求次数、关键帧数量、估算成本。

可折叠调试区：

- 本轮上传关键帧数量。
- 图片压缩后尺寸。
- 模型耗时。
- 估算 token 和估算成本。
- 限流命中原因。
- provider 错误码。

### 5.2 主要交互状态

系统至少包含以下状态：

- `idle`：待机，尚未开始监听。
- `permission-checking`：检查摄像头和麦克风权限。
- `listening`：正在监听用户说话。
- `transcribing`：正在产生语音转写。
- `capturing-frame`：正在截取关键帧。
- `thinking`：后端正在调用多模态模型。
- `responding`：AI 文本回复已返回。
- `speaking`：浏览器 TTS 正在播报。
- `rate-limited`：请求被成本控制策略拦截。
- `error`：权限、浏览器兼容性、网络或模型调用失败。

### 5.3 单轮对话流程

1. 用户点击开始。
2. 浏览器请求摄像头和麦克风权限。
3. 摄像头预览显示在左侧。
4. 用户说出问题。
5. 浏览器 Web Speech API 将语音转成文本。
6. 用户停止说话后，前端截取当前关键帧。
7. 前端根据画面变化检测和本轮问题决定是否补充关键帧，但总数不超过 3 张。
8. 前端压缩关键帧，并发送文本、关键帧和短期上下文到后端。
9. 后端执行限流、预算检查和请求校验。
10. 后端调用真实云端多模态模型。
11. 前端显示 AI 文本回复。
12. 浏览器 TTS 播报回复。
13. 前端更新会话历史、延迟和估算成本。

## 6. 系统架构

### 6.1 总体架构

系统分为前端采集交互层和后端模型代理层。

前端负责：

- 摄像头权限与实时预览。
- 麦克风权限与 Web Speech ASR。
- 关键帧截取、变化检测和压缩。
- 对话 UI、状态机和错误反馈。
- 浏览器 TTS 播报。
- 会话内短期上下文管理。
- 请求统计和成本估算展示。

后端负责：

- 接收文本、关键帧和短期上下文。
- 校验请求 schema。
- 执行服务端限流和预算限制。
- 调用云端多模态模型。
- 屏蔽模型供应商差异。
- 返回 AI 文本、延迟、估算 token、估算成本和错误信息。

云端模型负责：

- 理解用户问题。
- 理解关键帧视觉内容。
- 结合短期上下文回答。
- 在不确定时表达不确定，并给出可执行的画面调整建议。

### 6.2 数据流

```text
摄像头视频流
  -> 前端实时预览
  -> 用户说完后截取关键帧
  -> 图片压缩
  -> 后端请求

麦克风音频
  -> 浏览器 Web Speech API
  -> 用户文本
  -> 后端请求

用户文本 + 关键帧 + 短期上下文
  -> 后端限流与预算检查
  -> 多模态 provider
  -> 云端模型
  -> AI 文本回复
  -> 前端展示
  -> 浏览器 TTS 播报
```

### 6.3 Provider 设计

后端应通过 provider 适配层调用模型，而不是把供应商 SDK 直接写进业务流程。

建议抽象：

- `MultimodalProvider`：接收文本、图片和上下文，返回文本回复、token 估算、耗时和原始错误。
- `ASRProvider`：后续云端 ASR 预留，首版默认不启用。
- `TTSProvider`：后续云端 TTS 预留，首版默认不启用。

首版默认 provider 使用 OpenAI-compatible 接口形态，通过环境变量配置：

- `MODEL_BASE_URL`
- `MODEL_API_KEY`
- `MODEL_NAME`
- `MODEL_TIMEOUT_MS`
- `MODEL_MAX_OUTPUT_TOKENS`

前端不得持有模型 API Key。

## 7. 视觉理解准确性设计

首版不宣称通用视觉准确率，而用可演示任务定义成功。

### 7.1 场景描述

用户问“你看到了什么”时，AI 应能描述画面中 2-4 个主要可见元素，例如桌面、纸张、杯子、屏幕、键盘、手持物体等。

成功标准：

- 能覆盖主要可见元素。
- 不编造明显不存在的物体。
- 能区分“确定看到”和“可能是”。

### 7.2 指代追问

用户在同一会话中追问“左边那个是什么颜色”“它旁边还有什么”时，AI 应结合短期上下文和当前关键帧回答。

成功标准：

- 能理解“它”“左边那个”“刚才那个”等常见指代。
- 在画面发生变化时优先相信当前关键帧。
- 当上下文和画面冲突时说明不确定。

### 7.3 不确定性表达

当画面过暗、模糊、遮挡、文字太小或关键区域不在画面中时，AI 应明确说明看不清，而不是硬猜。

成功标准：

- 使用“看不清”“不确定”“需要更近一点”等表达。
- 给出可执行建议，例如“请把物体靠近摄像头”“请提高光线”“请让文字正对镜头”。
- 不输出高风险判断。

## 8. 语音交互自然度与流畅性设计

### 8.1 交互指标

首版语音体验按以下指标验收：

- 用户点击开始后，可以连续用语音发起多轮问题。
- 语音转写结果在用户说完后快速出现在对话区。
- 普通轮次目标体验为说完后 2-5 秒内看到 AI 文本回复。
- AI 文本先展示，TTS 随后播报，避免等待完整语音合成。
- 用户可以停止当前 TTS 播报。
- 系统明确展示当前状态。
- 失败时有明确提示，不静默失败。

### 8.2 浏览器兼容性边界

Web Speech API 在不同浏览器上的兼容性不同。首版推荐 Chrome 环境。

当浏览器不支持 Web Speech API 时：

- 不伪造语音转写。
- 明确提示当前浏览器不支持语音识别。
- 可提供手动输入作为调试辅助，但手动输入不计入语音交互能力的验收。

## 9. 端云协同与成本控制

### 9.1 想到的运营成本控制技巧

设计阶段考虑过以下成本控制技巧：

1. 浏览器 Web Speech API 优先，减少云端 ASR 成本。
2. 浏览器 TTS 优先，减少云端语音合成成本。
3. 用户触发式问答，避免后台持续视频理解。
4. 每轮默认上传 1 张关键帧，而不是上传连续视频流。
5. 画面变化明显或用户明确要求时，最多补充到 3 张关键帧。
6. 前端压缩图片尺寸和质量后再上传。
7. 服务端强制每轮关键帧数量上限。
8. 服务端限制每分钟多模态请求次数。
9. 服务端限制每会话最大轮数。
10. 文本追问优先复用上一轮视觉摘要，必要时才重新取帧。
11. 只传必要的短期上下文，避免 token 持续膨胀。
12. 限制模型最大输出长度。
13. 为不同任务选择不同模型，例如简单文本追问走低成本模型，复杂视觉问题走多模态模型。
14. 缓存最近一轮视觉摘要，减少重复看同一画面的成本。
15. UI 展示本轮和会话累计估算成本，提高用户对成本的感知。
16. 服务端配置单日调用上限，避免演示环境失控。
17. 对无效请求提前拦截，例如空文本、无摄像头帧、超大图片。
18. 超时快速失败，不让请求长时间占用资源。

### 9.2 首版实际采用的成本控制策略

首版设计决定采用以下策略：

- 浏览器 Web Speech API 作为默认 ASR。
- 浏览器 `speechSynthesis` 作为默认 TTS。
- 用户触发式问答，不做后台持续观察。
- 每轮默认 1 张关键帧，最多 3 张。
- 前端压缩关键帧。
- 服务端强制关键帧数量上限。
- 服务端强制请求频率上限。
- 服务端强制会话轮次上限。
- 服务端限制模型最大输出长度。
- 只发送必要的短期上下文。
- UI 显示请求次数、关键帧数量、延迟和估算成本。
- 模型 API Key 只放后端，避免前端滥用。

### 9.3 暂不采用的成本控制策略

以下策略有价值，但首版暂不采用：

- 多模型自动路由。
- 长期缓存跨会话视觉摘要。
- 复杂账单、用户余额和套餐系统。
- 云端 ASR/TTS 的精细化按需切换。
- 边缘端本地图像识别模型。
- 自动后台观察模式。

### 9.4 当前采用记录

截至 2026-06-13，前端已实际采用浏览器 Web Speech API 作为 ASR 入口，并已实际采用浏览器 `speechSynthesis` 作为 TTS 入口。语音转写结果复用现有 `/api/conversation-turn` 请求契约，不新增云端 ASR/TTS 调用，也不在前端保存原始音频。

本次验证证据：

- `apps/web/src/App.test.tsx` 覆盖独立麦克风入口、interim/final 转写、final 自动提交、TTS 播报、停止播报、TTS 结束后续听、ASR 不支持提示。
- `npm run test -w @ai-vision-voice-chat/web` 通过 17 个前端测试。
- `npm run typecheck -w @ai-vision-voice-chat/web` 通过。

## 10. 隐私与安全边界

首版默认不持久化原始媒体数据。

- 不录制完整视频。
- 不保存原始音频。
- 不落盘保存关键帧图片。
- 关键帧只在用户发起一轮问答时临时上传给后端。
- 后端只在请求生命周期内处理关键帧。
- 默认日志只记录文本、状态、耗时、错误码、关键帧数量和估算成本。
- 若未来需要保存截图、录音或录像，必须由用户显式触发，并在 UI 中说明保存范围。

安全限制：

- 不进行身份识别。
- 不进行敏感属性推断。
- 不输出医疗、法律、安全等高风险结论。
- 不在模型失败时伪造成功回复。
- 不在浏览器端暴露模型 API Key。

## 11. 错误处理

系统必须显式处理以下错误：

- 摄像头权限被拒绝。
- 麦克风权限被拒绝。
- 浏览器不支持 Web Speech API。
- 摄像头画面不可用。
- 用户语音为空或识别结果为空。
- 关键帧截取失败。
- 图片过大或压缩失败。
- 服务端限流。
- 会话预算超限。
- 模型调用超时。
- 模型 provider 返回错误。
- 网络断开。

错误提示应包含三部分：

- 发生了什么。
- 用户能做什么。
- 是否已经产生云端调用成本。

## 12. 验收标准

### 12.1 功能验收

- 用户可以打开摄像头并看到实时预览。
- 用户可以打开麦克风并完成至少一轮语音转写。
- 用户可以完成至少三轮视觉语音问答。
- 每轮请求包含用户文本和 1-3 张关键帧。
- 后端通过真实云端多模态模型返回文本回复。
- 前端使用浏览器 TTS 播报 AI 回复。
- 用户可以停止播报。
- UI 能显示当前状态、错误、延迟和估算成本。
- 超过关键帧数量、频率或会话轮次限制时，请求被后端拦截。

### 12.2 视觉理解验收

- 对常见桌面/室内场景，AI 能描述主要可见元素。
- 对同一会话内的简单指代追问，AI 能结合上下文回答。
- 对模糊、过暗或遮挡画面，AI 能表达不确定并给出调整建议。

### 12.3 语音体验验收

- 用户说完后，转写文本能显示在对话区。
- 普通轮次 2-5 秒内出现 AI 文本回复。
- TTS 播报不会阻塞下一轮状态恢复。
- 用户能中止当前播报。
- 权限失败、ASR 不支持、模型失败和限流都有明确提示。

### 12.4 成本控制验收

- 前端不会持续上传视频流。
- 每轮上传关键帧不超过 3 张。
- 后端对超限请求返回明确错误。
- UI 显示本轮和会话累计的请求统计。
- 模型 API Key 不出现在前端代码或浏览器网络请求中。

## 14. 风险与取舍

### 14.1 Web Speech API 兼容性

Web Speech API 的浏览器兼容性和稳定性不是完全可控。首版通过推荐 Chrome、显式错误提示和预留云端 ASR provider 降低风险。

### 14.2 多模态模型准确性

云端模型可能误识别画面。首版通过关键帧数量控制、不确定性提示和用户重新取帧来降低错误影响，但不承诺专业视觉检测能力。

### 14.3 延迟

端到端延迟由 ASR、图片压缩、网络和模型响应共同决定。首版用文本先显示、TTS 后播报的方式改善感知延迟。

### 14.4 成本

视觉模型调用成本高于纯文本。首版通过用户触发、关键帧上限、图片压缩、短上下文和服务端限流控制成本。

### 14.5 隐私

摄像头画面属于敏感输入。首版默认不持久化原始媒体，只在用户发起问答时临时上传关键帧，并在 UI 中展示当前采样状态。
